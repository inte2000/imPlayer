# iPlayer 开发日志 20260420



​         这一期的任务是让 iPlayer 支持外部解码器插件。主要的工作有几部分，首先是定义外部插件的接口，这部分是我手工做的，就是动态库的导出函数定义。其次是插件配置和管理，包括配置文件的处理。还有就是解码器配置，当多个解码器都支持同一种媒体文件类型时，可以选择优先使用某种解码器，当然，还包括增加一个具体的插件解码器项目 libsnd_decoder。本期的主要工作我也录制的相关的视频，就是具体的操作过程以及大模型的反馈，本篇最后有视频链接。



## 1 搭建 libsnd_decoder 项目

​        这一步就是添加一个 libsnd_decoder 动态库项目，使其具备必要的导出函数，让后续的外部解码器插件管理部分能够编译、测试，最后我们再实现这个解码器，让 iPlayer 能加载并使用这个插件。

​        作为一个动态库，要导出哪些函数才能具备解码器应有的功能？这块说实话，要让 AI 生成我想要的样子不是不可以，但是可能不如我自己做的快。于是我花了十几分钟时间，写好了 PlugMain.h 和 PlugMain.cpp 两个文件，都是函数声明，以及空的函数体，并且也包含了必要的头文件，主要是减少 AI 任务的复杂度。我首先选择大模型用 GLM 4.7，写好了一大段任务说明，在视频的第一部分可以看到，生成的 libsnd_decoder 可以编译，但是漏掉了动态库的导出函数，其实在 Plan 的时候，我还看到屏幕上它输出了与导出函数有关的计划，但是在实施时没做。我想可能的原因有两个，一个是 Plan 完成之后，我没有切换到 Build 模式，就直接让它执行，它有点糊涂了。因为 Agent.md 控制的很严格，Plan 模式不能执行任何修改，所以它最后只是把计划又输出了一遍，可能是在此时把这个细节搞丢了，在视频中也有这个过程。第二个原因可能是我在任务说明中没有明确说明用 def 文件定义导出符号，所以我再第二次使用 GPT 5.3 code 的时候，加了一句说明，所以最终用 GPT 的输出作为结果。

​        视频的第二部分就是切换到 GPT 后又做了一遍，并保留了这一次的修改。并不是说 GPT 比 GLM 强，我在工作的时候，两个模型都用，我觉得输出是否满意主要还是看我的控制。不吹不踩，目前我主要用的编程大模型就是 GPT 和 GLM。可以编出 libsnd_decoder.ipdplus 插件文件后，就先不管了，接着开始后续的工作，就是插件管理。

## 2 添加 DecoderDllWrapper

​        使用动态库麻烦的一点就是对模块句柄的管理，其实就是最后别忘了 Free 句柄。大部分情况下都是用一个包裹类，管理句柄的同时提供一点导出函数的封装。C++ 不像有些动态语言，可以动态操作类型，所以此类包裹类都很无趣，大都是整体上看是重复，但是细节又各不相同的重复，所以这个事情交给 AI 是最合适的。

​        这种工作没有难度，AI 完成的一般都还可以，这部分在视频中也有体现，大模型正确生成了这个包裹类，并且编译一次通过。至于功能是否正确，后面用到的时候再说。



## 3 添加 PluginsConfig

​        插件的配置需要持久化保存在磁盘上，所以需要一个保存和加载插件的步骤。这里要做的事情就是将某种内存结构存储的插件信息保存到 json 文件中，并且能从 json 文件中还原出相应的内存结构。如果仅仅是告诉大模型我需要加载和保存插件信息，大模型大概率就是给出两个函数，类似这样：

```
bool LoadPlusinConfigFile(const std::string& filename, std::vector<PlusinConfig>& plusItems);
bool SavePlusinConfigFile(const std::string& filename, const std::vector<PlusinConfig>& plusItems);
```

肯定是功能正确的，这一点不用怀疑，但问题是大模型的代码也就这样，你不加以引导，强调代码的可测试性，最终得到的就是可用的代码，至于其他方面，别想太多。所以我将任务描述改成这样，希望 GPT 能够直接给出我希望的结果：

```
这是插件配置文件的格式：

{
    "plugins": [
        {
            "hostfile": "libsnd_decoder.ipdplus",
            "name": "libsndfile decoder",
            "publisher": "iPlayer Group",
            "type": "Decoder"
        }
    ]
}

请定义一个与之相适应的数据结构 PluginConfig 存储插件信息，并实现加载和保存插件配置信息的函数：

bool LoadPluginConfigFile(const std::string& filename, std::vector<PluginConfig>& plusItems);
bool SavePluginConfigFile(const std::string& filename, const std::vector<PluginConfig>& plusItems);

为了更好的可测试性，请将 LoadPluginConfigFile 函数的实现分成两部分，第一部分从文件加载到 Json 类型，第二部分实现从 Json 到 std::vector<PluginConfig> 的解析，第二部分单独封装一个函数，并在 test 项目中添加对这个函数的单元测试代码。 对 SavePluginConfigFile 函数也是一样，将从 const std::vector<PluginConfig> 到 Json 对象的部分单独封装一个函数，同样在 test 项目中添加对这个函数的单元测试代码。
```

​        只要你提出来了，并让大模型理解了，大模型的输出结果还是令我满意的。它将 Load 和 Save 函数都拆解为两个步骤，并用形象的名称命名了这些函数，最后，test 中的测试用例也是一次生成。关于测试代码，只要生成的代码没有编译错误，我就不做任何修改。test 的代码都是大模型原汁原味的输出，仅供研究和参考，是否起到测试作用，我不太关心。因为 iPlayer 的原则就是尽量让 AI 生成，能跑就行。


## 4 添加 PluginsMgmt

​        在 DecoderFactory 对象中，内置插件以 DecoderMapItem 结构存在，DecoderMapItem::Creator 可以直接创建一个 CAudioDecoder 对象。外部解码器插件纳入 DecoderMapItem 之前，需要一个将 PluginConfig 中的简单描述经过加载、验证，转换成 DecoderMapItem 的过程，这个过程计划分两步进行。第一步是将 PluginConfig 中 dll 文件加载进来，得到 CDecoderDllWrapper，必要时需要查询相关的信息。

        所以这里向 GPT 提出要两个函数的要求，一个用于从配置文件中加载 CDecoderDllWrapper，用在 iPlayer 启动时加载插件的流程。另一个是直接从 dll 文件加载 CDecoderDllWrapper，此时需要调用动态库的 GetPluginInformation 函数获取一些信息。对这种情况我的习惯是直接给出函数声明，让 AI 填代码，这样对输出的预期更有确定性。
